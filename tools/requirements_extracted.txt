ТЗ
Цей документ описує обов’язкові вимоги до Android-застосунку, який розробник повинен передати як готовий проєкт.
Мова проєкту: Kotlin
UI: Jetpack Compose + Material 3
DI: Koin
Min SDK: 24+ (якщо не погоджено інакше)
Target SDK: 36
Цільова платформа: Android (без iOS, без KMP)
applicationId повинен починатися з com.
Приклад: com.clientname.appname
Готовий проєкт має відкриватися і збиратися в Android Studio без додаткових ручних кроків.
1. .gitignore та секрети
У корені проєкту має бути .gitignore, який виключає:
.idea, *.iml
усі build/ каталоги
ключі підпису і локальні приватні файли
тимчасові налаштування розробника
У репозиторій не можна додавати:
API ключі у відкритому вигляді
токени доступу
паролі та інші секрети
2. Архітектура і структура коду
Архітектура повинна бути простою, читабельною і підтримуваною. Не потрібно штучно ускладнювати проєкт багатошаровими патернами, якщо цього не вимагає реальна логіка застосунку.
У межах одного модуля app код має бути розділений по пакетах за призначенням:
Опис пакетів:
ui/
Composable-екрани
ViewModel-и
стани UI (data class-и типу UiState)
навігація між екранами
data/
мережеві виклики (API клієнти, сервіси)
локальне збереження даних (Room, DataStore)
репозиторії, які інкапсулюють доступ до даних
Важливо: мережеві запити та доступ до бази даних не повинні викликатися напряму з Composable.
di/
Koin модулі з оголошенням залежностей:
клієнт мережі
репозиторії
локальне сховище
ViewModel-и
utils/
дрібні допоміжні речі (наприклад, форматування дати, логер-обгортка і т.д.)
Не потрібно вводити окремий domain/ шар і usecase, якщо вони не додають реальної цінності. Але код всередині ViewModel не має бути “свалкою”: бізнес-логіка має бути хоча б винесена в окремі класи/репозиторії.
3. Ресурси (res/)
Усі ресурси повинні бути організовані і названі за правилами.
Зображення (растрові: PNG/WebP)
Папка: res/drawable
Формат імені: img_&lt;опис&gt;.png
Приклад: img_user_placeholder.png
Іконки (вектор .xml)
Формат імені: ic_&lt;опис&gt;.xml
Приклад: ic_settings.xml
Шрифти
Папка: res/font
Формат імені: &lt;fontname&gt;_&lt;style&gt;.ttf
Рядки (тексти інтерфейсу)
Файл: res/values/strings.xml
Формат ключів: lower_snake_case
Приклад:  &lt;string name=&quot;error_no_connection&quot;&gt;No internet connection&lt;/string&gt;
У UI не можна хардкодити тексти напряму в Text(&quot;...&quot;), якщо це текст, видимий користувачу. Потрібно використовувати stringResource(R.string.some_key).
Інші файли
Анімації (наприклад, Lottie .json), звуки тощо — у res/raw
Ім’я для анімацій: anim_&lt;опис&gt;.json
Наприклад: anim_success.json
Якщо потрібної папки немає — її треба створити.
4. Тема, токени дизайну та стилі
4.1. Загальна ідея
Застосунок повинен мати єдину дизайн-систему на базі токенів, яка передається через CompositionLocal.
Це означає:
ми НЕ тягнемо кольори напряму через Color(0xFF...)
ми НЕ задаємо розмір шрифту напряму в Text
ми читаємо все через наш об’єкт LocalAppTheme
4.2. LocalAppTheme
У пакеті theme/ має бути оголошений публічний доступ до теми приблизно в такому вигляді:
Імена типів (ColorTokens, TypographyTokens, ShapeTokens, DimensionTokens) можуть відрізнятися, але сенс повинен бути таким:
є дата-класи з токенами;
є CompositionLocal для кожної групи токенів;
є фасад LocalAppTheme, через який ці токени читаються.
4.3. Приклад використання в UI
Всередині будь-якого Composable заборонено писати прямо Color(0xFF...) або fontSize = 14.sp.

5. Навігація
Навігація між екранами повинна бути реалізована за допомогою Navigation Compose.
Усі маршрути екранів мають бути оголошені централізовано, наприклад у пакеті ui/navigation.
Дані між екранами потрібно передавати через прості типи (id, url, token, tabName тощо), а не через великі об’єкти.
Стан екрану відновлюється у ViewModel на основі отриманих параметрів.

7. Робота з мережею
Мережеві виклики повинні виконуватися через один централізований HTTP-клієнт (наприклад, Ktor Client).
Вимоги:
Налаштування клієнта (таймаути, логування, заголовки тощо) має бути винесено в окремий провайдер, а не дублюватися.
Для кожної групи запитів створюється сервіс (наприклад, AuthApi, ProfileApi) замість того, щоб виконувати HTTP напряму з ViewModel.
ViewModel працює з репозиторієм, репозиторій вже звертається до цих сервісів.
URL-адреси, ключі доступу та інші чутливі значення не можуть бути захардкожені напряму в коді запиту.
8. Локальні дані
Room
Якщо застосунок зберігає локальні дані:
Використовувати Room.
DAO, Entity та база (AppDatabase) мають знаходитись у пакеті data/db.
ViewModel не повинна напряму оперувати Room-сутностями. Репозиторій відповідає за перетворення їх у моделі, зручні для UI.
DataStore
Для простих налаштувань користувача, токенів сесії, прапорців.
9. Конфігураційні значення і ідентифікатор застосунку
Не можна в коді писати значення пакета вручну, наприклад:
Якщо потрібно знати package name / applicationId у рантаймі (для доступу до ресурсів, формування префіксів, deep links тощо), треба отримувати його з BuildConfig.APPLICATION_ID.
Це правило обов’язкове.
applicationId оголошується один раз у Gradle-конфігурації модуля app, і більше ніде не дублюється руками в коді.
10. Назва застосунку (App name) та збірка з оптимізаціями
Назва застосунку не може бути захардкоджена в коді або в UI.
Вимоги:
Назва застосунку повинна бути задана в одному місці
При зміні назви в цьому місці вона повинна автоматично оновлюватися у всьому застосунку
Заборонено:
хардкодити назву застосунку в Text(&quot;...&quot;)
використовувати рядкові літерали з назвою застосунку в коді
дублювати назву в декількох ресурсах без необхідності
Правильний підхід:
Основна назва застосунку зберігається у res/values/strings.xml, наприклад:  &lt;string name=&quot;app_name&quot;&gt;Client App&lt;/string&gt;
11. Підтримка minify та shrinkResources
Застосунок обовʼязково повинен коректно працювати з увімкненими оптимізаціями збірки.
Проєкт має збиратися і працювати з:
minifyEnabled trueshrinkResources true
Не допускається падіння застосунку у release-збірці
Якщо використовується:
reflection
serialization
DI
сторонні SDK
JSON-моделі
Room / Ktor / DataStore
— необхідні proguard-rules.pro повинні бути додані до проєкту.
